## 二、核心問題分析

### 問題 1：medications 表嚴重過時
- 現有 204 筆 vs ISAF 9,094 筆（**97.8% 數據缺失**）
- 6 欄位 vs ISAF 26 欄位（缺少多語、藥廠、ATC 分類等關鍵欄位）
- 欄位命名不一致（`name` vs `product_name_zh`，無 `generic_name` 等效欄位）

### 問題 2：ISAF 數據尚未整合
- CSV/JSON 游離於資料庫之外，前端無法實時搜尋
- `lookupController.js` 只能搜尋 MySQL，無法跨接 ISAF 數據

### 問題 3：ICD-10 表結構穩定但資料未更新
- 表結構 OK（8 欄，三語）
- 但 `init-data.js` 只 seed 28 筆，完整數據（8,989 行）在 CSV，需確認是否已全量匯入 MySQL

## 三、重構方案選型

### 方案 A：原地擴展 `medications` 表（推薦）
- **做法：** ALTER TABLE，加入 ISAF 所有欄位（三語 + ATC + 藥廠等）
- **優點：** SQL 查詢簡單，無跨表 JOIN
- **缺點：** 欄位數量多（從 6 → 28+），部分欄位（如 `active_ingredients_en`）對臨床用戶無直接價值
- **風險：** 低（純 Migration，不動代碼邏輯）

### 方案 B：新建 `isaf_medicines` 表
- **做法：** 保留原 `medications` 表不變，新建 `isaf_medicines` 表
- **優點：** 隔離關注點，原有功能不變
- **缺點：** 前端需調兩個 API（未來有「本地藥物 + ISAF 統一搜尋」需求時困難）
- **風險：** 中（未來介面整合成本高）

### 方案 C：完全替換 `medications` 表
- **做法：** DROP 舊表，CREATE 新表，全量匯入 ISAF 數據
- **優點：** 結構最乾淨
- **缺點：** 丟失所有歷史 ID，外鍵斷裂（`prescription_medications` 等有 FK）
- **風險：** 高（創傷性大手術）

## 四、推薦方案：方案 A（原地擴展）

### 4.1 理由
1. **業務連續性：** 不破壞現有 FK 關聯（`prescription_medications.medication_id` 等）
2. **漸進遷移：** 可以先加欄位，再慢慢遷移數據
3. **代價最低：** 純 ALTER TABLE，不回頭動 Controller

### 4.2 新 `medications` 表結構

```sql
ALTER TABLE medications;

-- 識別與名稱（保留原 id 體系，新增 ISAF 映射）
ADD COLUMN isaf_mednbr      VARCHAR(20)   DEFAULT NULL COMMENT 'ISAF 藥物編號（對應 mednbr）',
ADD COLUMN product_name_zh VARCHAR(255) DEFAULT NULL COMMENT '產品名稱（中文）',
ADD COLUMN product_name_pt VARCHAR(255) DEFAULT NULL COMMENT '產品名稱（葡文）',
ADD COLUMN product_name_en VARCHAR(255) DEFAULT NULL COMMENT '產品名稱（英文）',
ADD COLUMN product_name   VARCHAR(255) DEFAULT NULL COMMENT '產品名稱（本地語言，別名）',

-- 學名/成份（保留原有 generic_name，新增多語）
ADD COLUMN generic_name_en VARCHAR(500) DEFAULT NULL COMMENT '學名（英文）',
ADD COLUMN generic_name_pt VARCHAR(500) DEFAULT NULL COMMENT '學名（葡文）',

-- 劑型與途徑（新增多語，保留原 dosage/route 別名）
ADD COLUMN pharmaceutical_form_zh VARCHAR(100) DEFAULT NULL COMMENT '劑型（中文）',
ADD COLUMN pharmaceutical_form_en VARCHAR(100) DEFAULT NULL COMMENT '劑型（英文）',
ADD COLUMN route_of_administration_zh VARCHAR(50)  DEFAULT NULL COMMENT '給藥途徑（中文）',
ADD COLUMN route_of_administration_en VARCHAR(50)  DEFAULT NULL COMMENT '給藥途徑（英文）',

-- 活性成份（保留原有）
ADD COLUMN active_ingredients_zh TEXT DEFAULT NULL COMMENT '活性成份（中文，| 分隔）',
ADD COLUMN active_ingredients_en TEXT DEFAULT NULL COMMENT '活性成份（英文，| 分隔）',

-- 分類（新增）
ADD COLUMN legal_classification_zh VARCHAR(100) DEFAULT NULL COMMENT '法律分類（中文）',
ADD COLUMN atc_classification_zh  VARCHAR(255) DEFAULT NULL COMMENT 'ATC 分類（中文）',
ADD COLUMN atc_classification_en  VARCHAR(255) DEFAULT NULL COMMENT 'ATC 分類（英文）',

-- 藥廠與經銷商（新增）
ADD COLUMN manufacturer_zh VARCHAR(255) DEFAULT NULL COMMENT '藥廠（中文）',
ADD COLUMN manufacturer_en VARCHAR(255) DEFAULT NULL COMMENT '藥廠（英文）',
ADD COLUMN distributor_code VARCHAR(50)  DEFAULT NULL COMMENT '經銷商代碼',
ADD COLUMN distributor_zh   VARCHAR(255) DEFAULT NULL COMMENT '經銷商（中文）',

-- 狀態欄位
ADD COLUMN source  ENUM('local', 'isaf') DEFAULT 'local' COMMENT '數據來源',
ADD COLUMN is_active BOOLEAN DEFAULT TRUE COMMENT '是否啟用';

-- 索引優化
ADD INDEX idx_medications_isaf (isaf_mednbr),
ADD INDEX idx_medications_source (source),
ADD INDEX idx_medications_name_zh (product_name_zh),
ADD INDEX idx_medications_name_en (product_name_en);
```

> **注意：** 原有欄位 `name`, `generic_name`, `dosage`, `route` 保留作為本地藥物的別名，並填入 ISAF 數據時從 `product_name_zh` / `generic_name` / `pharmaceutical_form_en` / `route_of_administration_zh` 自動映射填充，確保向後兼容。

### 4.3 ICD-10 表無需變更
- 結構已支援三語（tc/en/pt），符合需求
- 需確認 `import_icd10_to_db.cjs` 是否全量執行（8,989 行）

## 五、Migration 策略

### 5.1 三階段遷移

| 階段 | 操作 | 風險 |
|------|------|------|
| **Phase 1** | ALTER TABLE：新增所有 ISAF 欄位 + 索引 | 低（只加不减） |
| **Phase 2** | 寫入腳本：CSV → MySQL（全量 9,094 筆 Upsert） | 低（Upsert 不刪除） |
| **Phase 3** | 修改 `lookupController.js`：SELECT 新增所有欄位 | 中（需同步更新前端） |

### 5.2 向後兼容
- 新舊欄位同時存在，`name` = `product_name_zh` 別名
- 前端目前只查 `id, name, generic_name, dosage, route` — 這些欄位繼續有效
- API 回傳可同時包含新舊欄位，前端逐步遷移
